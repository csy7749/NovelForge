# NovelForge 项目介绍

本文档面向第一次接触 `NovelForge` 的开发者或协作者，目标不是替代 [README](./README.md)，而是帮助你快速建立对项目的整体认知，方便后续继续深入提问和查代码。

## 1. 项目定位

`NovelForge` 是一个面向长篇小说创作的 AI 辅助系统。它的核心思路不是单纯做“文本补全”，而是把小说创作拆成一套结构化、可追踪、可自动化的内容系统。

项目的几个核心设计理念：

- 以“卡片”为基本创作单元，将世界观、角色、分卷、章节等内容拆成结构化数据
- 以 `Schema JSON` 约束卡片内容结构，让前端表单、AI 生成、工作流节点共享同一份数据契约
- 以“上下文注入”把相关卡片、知识和历史内容组织成 AI 可理解的输入
- 以“工作流”承载自动化操作，将卡片生成、卡片联动、项目初始化等流程代码化
- 以“知识图谱 / 关系数据”辅助维持长篇创作中的一致性

一句话理解：它不是一个单纯的编辑器，而是一套“结构化创作 + AI 生成 + 自动化编排”的小说创作引擎。

## 2. 总体架构

项目由前后端两部分组成：

- `backend/`
  - FastAPI + SQLModel
  - 负责 API、数据模型、AI 调用、Schema 组装、工作流执行、上下文解析等核心逻辑
- `frontend/`
  - Electron + Vue 3 + TypeScript + Pinia + Element Plus
  - 负责桌面端 UI、卡片编辑体验、Schema 可视化构建、工作流编辑器、灵感助手等交互

运行方式：

- 后端默认运行在 `54321` 端口
- 前端桌面应用通过 Electron 承载，渲染层是 Vue 单页应用
- 前端默认通过 `/api` 前缀访问后端

## 3. 你应该先理解的五个核心概念

### 3.1 项目 Project

项目是创作容器。一部小说通常对应一个 `Project`，其下挂载整棵卡片树、工作流运行记录、伏笔、知识等数据。

### 3.2 卡片类型 CardType

卡片类型定义“某一类卡片的默认行为”，例如：

- 作品标签
- 世界观设定
- 角色卡
- 分卷大纲
- 章节正文

卡片类型上会定义：

- `json_schema`：该类型的默认结构
- `ai_params`：默认 AI 参数
- `editor_component`：前端使用哪个编辑器
- `default_ai_context_template`：默认上下文注入模板

### 3.3 卡片 Card

卡片是实际内容实例。卡片构成树形结构，可以有父子关系。

卡片上最重要的字段有：

- `content`：实际内容，JSON 结构
- `json_schema`：实例级结构覆盖，可为空
- `ai_params`：实例级 AI 参数覆盖，可为空
- `card_type_id`：所属类型
- `parent_id`：树形父节点

可以理解为：

- `CardType` 负责定义默认规则
- `Card` 负责保存具体内容

### 3.4 Schema

Schema 是整个项目最关键的中枢之一。

它的作用不只是前端表单定义，而是统一约束：

- 卡片该有哪些字段
- 字段的数据类型是什么
- 前端应该如何渲染
- AI 应该生成什么结构
- 工作流节点如何校验结构化输出

这里用的是 JSON Schema 风格，但项目在“结构构建器”层又做了少量更易理解的抽象，例如：

- 基础类型：`string` / `number` / `integer` / `boolean`
- `relation(embed)`：实际落地为 `$ref`
- `tuple`：实际落地为 `array + prefixItems`

### 3.5 工作流 Workflow

工作流是自动化执行层。

它可以用于：

- 新建项目时自动初始化一批卡片
- 某张卡片保存后自动触发联动
- 将一个结构化结果落地为多张卡片
- 组合多个 AI 节点完成复杂生成流程

当前项目中的工作流是“代码式工作流”，定义保存在数据库的 `definition_code` 中，而不是单纯的可视化节点 JSON。

## 4. 核心业务链路

如果你想快速把握项目，请优先理解下面这条主线：

### 4.1 卡片类型定义默认结构

内置卡片类型在启动阶段初始化，并根据内置 Pydantic 模型生成默认 `json_schema`。

例如：

- 角色卡会对应一个结构化模型
- 分卷大纲会对应一个结构化模型
- 章节正文会对应一个更适合文本编辑的模型

### 4.2 前端按 Schema 渲染表单

前端会加载 schema，并将其转换为动态表单。对于通用卡片，默认使用 `ModelDrivenForm` 或通用编辑器来渲染。

如果某个类型配置了专用编辑器，例如标签编辑器、章节正文编辑器，则走定制组件。

### 4.3 AI 生成按 Schema 输出

AI 生成时，并不是直接让模型“随便写”，而是会把目标 schema 作为输出契约：

- 后端组装完整 schema
- 根据需要过滤掉 `x-ai-exclude` 字段
- 构造动态 Pydantic 模型
- 校验 AI 输出

这使得结构化生成更可控。

### 4.4 卡片保存触发工作流或事件联动

卡片保存后，会通过事件系统触发后续逻辑，例如：

- 更新关联信息
- 触发工作流
- 自动创建下游卡片

这也是项目强调“事件驱动”和“低耦合”的原因。

## 5. 目录导读

### 5.1 后端目录

`backend/app/` 是后端主目录，重点模块如下：

- `api/`
  - HTTP 路由入口
  - 典型接口包括项目、卡片、AI、工作流、提示词、知识、关系图等
- `db/`
  - SQLModel 数据模型和数据库会话
- `schemas/`
  - Pydantic 请求/响应模型
  - 也包含很多“内置内容模型”，这些模型可以直接导出为 JSON Schema
- `services/`
  - 业务核心
  - 包括卡片服务、Schema 服务、AI 服务、工作流引擎、上下文服务等
- `core/`
  - 应用配置、启动流程、中间件、事件系统
- `bootstrap/`
  - 启动时初始化内置数据，如默认卡片类型等

### 5.2 前端目录

`frontend/src/renderer/src/` 是 Vue 渲染层主目录，重点如下：

- `views/`
  - 页面级入口
  - 主要是 `Dashboard`、`Editor`、`IdeasHome`、`CodeWorkflowEditor`
- `components/`
  - 各类 UI 组件
  - 包含卡片编辑器、动态表单、设置面板、工作流编辑器、助手面板等
- `stores/`
  - Pinia 状态管理
- `services/`
  - 前端逻辑服务，如 schema 字段解析
- `api/`
  - 后端接口请求封装
- `utils/`
  - 辅助方法，如 schema 构建器转换逻辑

### 5.3 Electron 目录

`frontend/src/main/` 是 Electron 主进程：

- 创建窗口
- 注册 IPC
- 管理桌面应用壳层能力

`frontend/src/preload/` 是预加载脚本，用于安全暴露桌面能力给渲染进程。

## 7. Schema 相关设计

Schema 是你后续大概率会反复接触的模块，建议优先建立清晰认知。

### 7.1 Schema 的来源

来源分两类：

- 内置内容模型
  - 由后端 Pydantic 模型通过 `model_json_schema()` 生成
- 卡片类型 / 卡片实例自定义结构
  - 由用户在 Schema Studio 中通过结构构建器维护

### 7.2 Schema 的层级

大致分三层：

1. 全局内置 schema 仓库
2. `CardType.json_schema` 类型默认结构
3. `Card.json_schema` 实例覆盖结构

卡片真正使用的是 `effective_schema`，规则通常是：

- 卡片实例有自定义 schema，就优先用实例
- 没有实例 schema，就回退到类型 schema
- 然后后端再补齐 `$defs` 和被引用结构

### 7.3 Schema 的作用范围

同一份 schema 会被多处复用：

- 动态表单渲染
- Schema Studio 可视化编辑
- AI 结构化生成
- 工作流节点输出校验

这也是项目“Schema-first”设计的核心价值。

## 8. AI 系统相关设计

AI 在这个项目里不是单个接口，而是几层能力叠加：

- LLM 配置管理
  - 管理不同模型供应商、模型名、API 地址、密钥等
- Prompt 管理
  - 管理内置和自定义提示词模板
- 结构化生成
  - 按 schema 输出 JSON 结构
- 指令流生成
  - 以字段级、路径级操作的方式逐步生成内容
- 灵感助手
  - 提供跨卡片、跨项目的对话与辅助创作能力

从产品角度看，项目里至少有两种典型 AI 使用方式：

- “当前这张卡片怎么写”
  - 更偏局部生成、结构化输出、直接落卡
- “围绕多个卡片持续讨论和推演”
  - 更偏灵感助手、引用上下文、工具调用

## 9. 工作流系统相关设计

工作流是项目的第二条主线。

### 9.1 工作流本质

它是一个代码式自动化引擎，支持：

- 节点输入输出 schema
- 运行状态追踪
- 节点级进度展示
- 暂停 / 恢复
- 运行记录保留策略

### 9.2 工作流常见用途

- 项目初始化模板
- 卡片落子
- 多步 AI 编排
- 拆书流程
- 内容批处理

### 9.3 为什么它重要

很多“保存一张卡，自动生成若干张下游卡片”的能力，本质都建立在工作流和事件系统之上，而不是写死在某一个页面或某一个 service 中。

## 10. 数据存储概览

项目当前的核心数据主要存储在 SQLite 中。

常见核心表包括：

- `Project`
- `CardType`
- `Card`
- `Prompt`
- `LLMConfig`
- `Workflow`
- `WorkflowRun`
- `NodeExecutionState`
- `Knowledge`
- `ForeshadowItem`

README 里还提到 Neo4j 可选能力，但当前仓库也已经实现了基于 SQLite 的关系图谱替代方案，因此并不是强依赖 Neo4j。

## 11. 工程规则与开发风格

项目明确强调以下工程方向：

- 低耦合
- 高内聚
- 事件驱动优先
- 插件化注册优先
- 配置集中管理
- API 契约单一事实来源
- 工作流改动必须走可校验闭环

简单说，这个项目不希望你把跨模块联动硬编码到单点逻辑里，而是鼓励：

- 通过事件触发联动
- 通过 schema 统一契约
- 通过注册机制扩展节点、初始化器、处理器
