## Context

当前仓库中的工具能力已经可用，但组织方式不统一：

- 助手工具主要通过 `@tool` 和局部依赖注入方式暴露
- 不同工具的返回结构存在差异
- 权限边界和确认策略更多是通过提示词约束，而非系统级配置
- 工具事件虽然已有部分流式输出，但没有形成稳定统一的契约

如果不先规范工具管线，后续多 agent、知识库注入和上下文调试都会被底层不一致放大。

## Goals / Non-Goals

**Goals:**

- 统一工具定义与注册方式
- 统一工具参数 schema 与结果协议
- 支持显式的工具权限范围和确认策略
- 提供稳定的工具调用事件流，便于前端展示和调试

**Non-Goals:**

- 不在本次变更中重写所有业务工具逻辑
- 不在本次变更中引入外部通用 MCP 框架替换现有后端
- 不在本次变更中做完整 GUI 工具市场或插件市场

## Decisions

### 1. 工具统一抽象为 definition + executor + policy

原因：

- 当前最缺的是“工具元信息”和“执行逻辑”的分离
- 这样既能兼容现有工具函数，也能承载权限和确认规则

### 2. 参数契约统一采用结构化 schema，而不是仅靠自然语言说明

原因：

- 多 agent 和前端工具展示都需要稳定参数格式
- 结构化 schema 能直接驱动校验、渲染和日志

### 3. 工具结果统一回到标准 result/event 协议

原因：

- 当前前端已经消费部分流式事件，但口径不稳定
- 统一协议后可直接支持工具历史、确认卡片、错误展示

### 4. 一期优先规范写入型和高频查询型工具

原因：

- 这些工具对数据安全和用户可解释性影响最大
- 先把高风险高频工具规范好，比全面铺开更实用

## Risks / Trade-offs

- [风险] 抽象层增加后，简单工具接入成本会上升  
  → Mitigation：保留轻量注册辅助函数，减少样板代码

- [风险] 旧工具与新协议并存一段时间会造成双轨  
  → Mitigation：分批迁移，并定义新旧桥接适配层

- [风险] 权限/确认策略若设计过重，会降低调用流畅度  
  → Mitigation：把高风险写入和低风险查询分级处理

## Migration Plan

1. 先定义工具模型、schema、result/event 协议
2. 为现有助手工具增加统一注册入口
3. 分批迁移高频工具
4. 接入 agent 级工具白名单
5. 清理旧的临时注册路径

回滚策略：

- 保留旧工具适配层，必要时可按入口切回原逻辑

## Open Questions

- 工具 policy 应存于代码、数据库还是配置文件？
- 前端是否需要直接消费工具 schema 来渲染确认界面？
- 是否要为工作流节点复用同一套工具定义模型？
