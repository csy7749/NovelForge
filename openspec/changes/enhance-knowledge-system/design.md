## Context

当前仓库已经有几种和知识相关但彼此分散的能力：

- `knowledge` 是简单的数据库条目 CRUD
- `memory` 侧偏向图谱事实与提取结果
- `context_service` 会装配一部分结构化事实进入模型上下文
- 还有若干 bootstrap 内置文本知识

这些能力加在一起，说明系统已经有“知识”的雏形，但缺少一个明确的知识系统模型，导致：

- 用户不知道什么该放在 knowledge，什么该放在 memory 或卡片
- AI 上下文注入缺少分层与可控策略
- 知识条目难以扩展为可维护、可检索的长期资产

## Goals / Non-Goals

**Goals:**

- 定义分层知识文档模型和核心类型
- 支持知识条目的注入模式与摘要策略
- 提供最小可用的检索层
- 明确 knowledge 与 memory、context、agent 的协作边界

**Non-Goals:**

- 不在本次变更中一次性交付完整向量数据库和 embedding 运行时
- 不在本次变更中重写现有 memory 图谱
- 不在本次变更中实现复杂外部文档同步平台

## Decisions

### 1. 知识文档采用分类型模型，而不是继续使用单一平铺文本条目

原因：

- 不同知识类型的注入和维护方式不同
- 分类型后，前端和 agent 才能合理选择知识来源

### 2. 注入模式与摘要能力作为知识文档的一等属性

原因：

- 当前上下文最大的问题之一是“什么该注入、注入到什么程度”
- 若没有注入策略，知识库会迅速变成新的上下文噪音源

### 3. 一期先做关键词/结构化检索，语义检索只预留接口

原因：

- 当前项目仍缺基础文档模型和注入边界
- 先解决“可管理、可解释、可检索”，再上 embedding 才稳

### 4. knowledge 与 memory 明确分工

原则：

- `memory` 负责动态事实、关系、运行中的故事状态
- `knowledge` 负责较稳定的设计文档、参考资料、技能说明、长期规则

## Risks / Trade-offs

- [风险] 如果类型划分过早，可能会让用户感到复杂  
  → Mitigation：一期只提供少量高价值类型，并给出清晰用途说明

- [风险] 没有语义检索时，大型知识库检索体验仍有限  
  → Mitigation：先把注入边界和摘要能力做扎实，为二期语义检索铺路

- [风险] knowledge 与 memory 边界模糊，可能出现重复录入  
  → Mitigation：在 spec 中明确两者职责，避免混用

## Migration Plan

1. 定义知识文档模型与类型
2. 为知识接口和存储增加类型、注入模式、摘要字段
3. 增加基础检索与过滤能力
4. 逐步接入 AI 上下文注入链路
5. 后续再评估语义检索和外部同步

回滚策略：

- 新字段与新接口以兼容方式增加，可退回到基础 CRUD 使用方式

## Open Questions

- 知识条目应优先存数据库、文件系统，还是双存？
- 摘要应由用户手写、系统生成，还是混合维护？
- skill 类知识是否要与未来工具/agent 配置建立强关联？
