## Context

当前仓库已经有一条较完整的上下文链路，但可观察性不足：

- 后端 `assemble_context` 已返回 `facts_subgraph` 和 `facts_structured`，可供写作或审核使用
- 前端 `contextResolver` 已支持卡片引用、事实子图引用、`kg:*`、`$self/$parent/$current` 等模板语法
- 卡片编辑器已经存在 `ContextDrawer` 入口，且生成、审核都依赖解析后的上下文文本

问题不在于“没有上下文”，而在于“上下文不可解释”：

- 用户只能看到结果文本，看不到每一段上下文来自哪张卡或哪类事实
- 开发者很难判断是模板写错、解析错、上下文缺失，还是被截断
- 后续如果引入多 agent、知识库注入或工具选择，缺少一层稳定的上下文观测面，会进一步放大调试成本

本次变更需要遵守当前工程约束：

- 保持最小增量，不提前引入大规模架构重写
- 不新增静默 fallback，来源缺失或解析失败要明确暴露
- 复用现有 `assemble_context` 与 `ContextDrawer`，避免新建平行入口
- 不把多 agent、知识库增强强行并入本次 MVP

## Goals / Non-Goals

**Goals:**

- 为上下文模板提供可视化的模板层、解析层、注入层三段式视图
- 为后端装配结果补充可追踪的来源元数据，支撑前端解释 UI
- 让用户能够定位上下文片段来自卡片、图谱事实、实体摘要还是当前卡片变量
- 让开发者能够判断上下文是否命中、是否为空、是否被截断
- 在不改变现有模型调用链的前提下提升生成与审核的可调试性

**Non-Goals:**

- 不在本次变更中实现完整多 agent 编排 UI
- 不在本次变更中引入新的 embedding、向量检索或知识库索引引擎
- 不在本次变更中重写上下文模板 DSL
- 不在本次变更中替换现有生成接口或工作流引擎
- 不在本次变更中实现复杂图形化知识图谱画布；一期只要求结构化可解释视图

## Decisions

### 1. 将上下文可视化挂载到现有 `ContextDrawer`，而不是新建独立页面

原因：

- 当前用户已经在卡片编辑和 AI 操作前通过上下文抽屉检查模板
- 可视化需求与“当前卡片、当前模板、当前生成动作”强绑定
- 接入面最小，能直接复用 `GenericCardEditor` 的现有上下文状态

备选方案：

- 新建全局“上下文工作台”页面：更适合二期横向排查，但会复制当前卡片和生成场景上下文，MVP 成本过高

### 2. 上下文可视化采用三层模型：模板层、解析层、注入层

定义：

- 模板层：展示原始模板与 token 高亮
- 解析层：展示每个 token 命中的结构化结果与来源元数据
- 注入层：展示最终发送给模型的纯文本上下文

原因：

- 这是用户和开发最关心的三种问题分界：写了什么、命中了什么、模型看到了什么
- 比直接展示最终文本更适合 debug
- 能自然兼容未来更多来源类型

备选方案：

- 只展示最终文本：实现最省，但无法解释命中来源
- 只展示结构化 JSON：对开发友好，但对写作者过重

### 3. 后端为上下文返回新增 trace 数据，而不是让前端自行二次推断

原因：

- `assemble_context` 已掌握图谱查询、实体摘要生成与截断逻辑
- 若由前端重算来源，会复制业务规则并产生前后端不一致
- 来源是否被截断、命中顺序、来源类别等信息应由单一后端源头产出

建议的 trace 单元至少包含：

- `kind`: 来源类别，如 `template_token`、`facts_relation`、`item_summary`、`concept_summary`、`card_reference`
- `label`: 用户可读标签
- `source_ref`: 来源对象标识，如卡片 id、实体名、token 名
- `preview`: 预览文本
- `count`: 命中数量
- `truncated`: 是否截断

备选方案：

- 前端根据 `facts_structured` 和模板字符串自行拼装 trace：实现快，但规则会分叉

### 4. 一期只要求结构化列表和标签视图，不强制图形关系图

原因：

- 当前 `facts_structured.relation_summaries` 已足够做 chips、列表和分组视图
- 图谱画布会明显增加布局、交互和性能成本
- 先把“可解释”做对，再决定是否做“图形化”

备选方案：

- 直接做关系图：视觉冲击更强，但难以在本次范围内稳定交付

### 5. 将多 agent、工具规范化、知识库增强作为本次设计约束，而不是交付内容

原因：

- 这三者会扩展上下文来源类型，但不是本次 change 的直接交付对象
- 本次 trace 和视图模型应预留可扩展字段，以便未来挂接 agent memory、tool outputs、knowledge snippets
- 若现在把范围一起做，会导致 proposal 失焦、tasks 失控

## Risks / Trade-offs

- [风险] 现有 `assemble_context` 返回结构较轻，补 trace 后接口模型会变复杂  
  → Mitigation：保持新增字段可选，并与现有 `facts_subgraph`/`facts_structured` 并存

- [风险] 模板解析逻辑目前集中在前端，若 trace 与前端解析粒度不一致，UI 可能出现两套口径  
  → Mitigation：一期先对后端装配结果和已支持 token 做有限对齐，不强求覆盖所有 DSL 角落语法

- [风险] 可视化信息过多会让写作者觉得面板太“工程化”  
  → Mitigation：默认优先展示简洁摘要，详细 trace 可折叠展开

- [风险] 如果来源缺失或解析失败，用户可能看到更多“报错信息”  
  → Mitigation：这是预期行为，必须显式暴露异常来源，避免 silent fallback

- [风险] 三层视图会增加前端状态复杂度  
  → Mitigation：将状态收敛在 `ContextDrawer` 和独立的上下文可视化子组件中，不扩散到全局 store

## Migration Plan

1. 先在 OpenSpec 中确认一期能力边界，只覆盖上下文可视化
2. 后端扩展上下文响应模型，返回可选 trace 数据
3. 前端在 `ContextDrawer` 中接入三层视图与来源展示
4. 在卡片编辑器中对生成、审核两种上下文入口做联调验证
5. 后续若要接入多 agent 或知识库片段，再另起 change 扩展来源类型

回滚策略：

- 若前端可视化体验不稳定，可移除 `ContextDrawer` 的新标签页或新面板组件
- 若 trace 接口导致兼容问题，可保留旧字段并临时关闭新 trace 渲染
- 本次不涉及数据库迁移，回滚不需要数据恢复

## Open Questions

- 模板层是否需要支持点击 token 后高亮对应解析结果，还是一期先做静态映射即可？
- 对于 `kg:*` 这类动态图谱 token，一期是否直接展示关系列表，还是只展示摘要数量？
- 审核场景和生成场景是否需要在同一视图中并列对比两个不同上下文模板？
- trace 数据是否要记录字符预算与截断前长度，作为后续上下文优化依据？
