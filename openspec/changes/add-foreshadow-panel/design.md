## Context

当前仓库已经具备伏笔领域的后端基础能力：

- `backend/app/api/endpoints/foreshadow.py` 提供 `suggest / list / register / resolve / delete`
- `backend/app/services/foreshadow_service.py` 已实现对应服务逻辑
- `frontend/src/renderer/src/api/ai.ts` 已有伏笔相关调用封装

但编辑器侧并未消费这些能力。当前前端右侧侧栏由 `Editor.vue` 管理，章节正文卡片会显示 `助手 / 参与实体 / 提取 / 大纲 / 审核结果` 多个 Tab，非章节卡片则只显示 `助手`。这意味着“伏笔”最适合以右侧面板的形式接入，而不是新建一个完全独立的页面或窗口。

本次变更需要遵守仓库现有工程约束：

- 保持最小变更，不引入无关重构
- 优先复用现有接口与状态流
- 不把业务规则重复写在多个位置
- 不在业务入口中硬编码跨域联动链路

## Goals / Non-Goals

**Goals:**

- 为项目提供明确可见的伏笔列表和操作入口
- 支持用户在 GUI 中手动登记伏笔，并在章节上下文中默认绑定当前章节
- 支持基于当前章节正文生成候选伏笔并批量登记
- 支持对现有伏笔进行回收、删除和定位跳转
- 以最小代价复用现有后端 API 和编辑器结构

**Non-Goals:**

- 不在本次变更中扩展 `ForeshadowItem` 数据模型字段
- 不在本次变更中新增自动工作流联动或 `card.saved` 自动登记
- 不在本次变更中新增独立的全局页面、菜单或 Electron 窗口
- 不在本次变更中替换现有 `frontend/src/renderer/src/api/ai.ts` 的 API 类型组织方式

## Decisions

### 1. 将伏笔能力挂载到编辑器右侧侧栏，而不是新增独立页面

原因：

- 当前右侧侧栏已经承载章节写作辅助能力，用户心智一致
- 伏笔登记与章节正文强相关，侧栏比全局页面更接近触发点
- 仅需改动 `Editor.vue` 和新增一个面板组件，破坏面最小

备选方案：

- 新增全局“伏笔工作台”页面：查看体验完整，但会复制项目切换、当前卡片上下文和导航逻辑，超出当前需求
- 放入顶部 Header 入口：适合作为二期全局汇总，不适合作为本次最小实现

### 2. 伏笔面板对所有激活卡片可见，但章节正文提供增强能力

原因：

- “查看项目全部伏笔”不应依赖用户必须先点到章节正文
- 当激活卡片是章节正文时，可以自然获得 `chapter_id` 和正文文本，用于默认绑定与候选建议
- 非章节卡片下仍可作为项目级查询面板使用

备选方案：

- 仅在章节正文显示：实现更简单，但会让“查看项目全部伏笔”入口过于隐蔽

### 3. MVP 直接复用现有后端接口，不新增后端字段和新端点

原因：

- 当前接口已覆盖 `list / register / resolve / delete / suggest`
- 需求核心缺的是前端可用性，而不是后端能力
- 不改表结构可避免迁移、类型同步和历史数据处理

备选方案：

- 立即扩展 `ForeshadowItem`，增加 `source_card_id / resolved_by_card_id / related_entities`：长期更完整，但超出本次 MVP

### 4. 面板组件使用局部状态 + 现有 store 辅助导航，不单独引入新的全局 store

原因：

- 伏笔面板状态局限在右侧侧栏，生命周期与编辑器强绑定
- 当前已有 `useCardStore` 和 `Editor.vue` 的 `handleJumpToCard` 可以复用
- 避免为单一面板引入新的全局复杂度

备选方案：

- 新建 `useForeshadowStore`：适合未来全局工作台，但当前会增加维护成本

## Risks / Trade-offs

- [风险] 现有 `ForeshadowItem.chapter_id` 语义较弱，只能表达关联章节，不能表达“由哪个条目回收”  
  → Mitigation：在 spec 和任务中明确这是 MVP 限制，后续另起变更扩展数据模型

- [风险] `suggest` 端点当前是启发式规则，不是高质量 AI 识别  
  → Mitigation：在面板中将其定位为“候选建议”，必须由用户勾选确认后登记

- [风险] 若右侧 Tab 再增加一个，章节编辑器信息密度会上升  
  → Mitigation：仅增加一个聚焦面板，不新增多层嵌套操作；保持与现有 Tab 命名和样式一致

- [风险] 非章节卡片下进行登记时，无法自动绑定章节  
  → Mitigation：允许创建项目级伏笔，不强制 `chapter_id`

## Migration Plan

1. 先交付 OpenSpec 变更文档并确认范围
2. 前端新增 `ForeshadowPanel.vue`，接入现有伏笔 API
3. 在 `Editor.vue` 右侧 Tab 中增加 `伏笔`
4. 完成基础交互验证：列表、登记、建议、回收、删除、跳转
5. 若后续需要更强追踪字段，另起 OpenSpec 变更扩展后端模型

回滚策略：

- 若交互不稳定，可仅移除右侧 Tab 挂载和新面板组件
- 由于本次 MVP 不改数据库结构，回滚不涉及数据迁移
