## Context

当前仓库已经有多条 AI 执行路径，但还没有统一的多 agent 抽象：

- `assistant_service.py` 同时承载标准工具模式、React 模式和纯对话降级
- `memory_service.py` 通过不同抽取器承担关系提取、动态信息提取等专职职责
- 工作流中已有若干 AI 节点与 `workflow_agent` 入口

这些事实说明项目并不是“没有多个智能角色”，而是缺少把它们显式建模为 agent 的一层编排结构。问题主要体现在：

- 单一助手入口的系统提示与工具集合持续膨胀
- 不同任务对上下文和工具权限的需求不同，但当前隔离不清
- 缺少统一的委派、状态追踪和结果回流协议

## Goals / Non-Goals

**Goals:**

- 定义多 agent 的基础数据模型与执行流程
- 支持主 agent 按任务类型委派给专职 agent
- 为每类 agent 限定职责、工具集合和默认上下文策略
- 保持与现有助手和工作流体系兼容，逐步接入

**Non-Goals:**

- 不在本次变更中实现完整的可视化 agent 工作台
- 不在本次变更中强制所有 AI 调用都必须经过多 agent
- 不在本次变更中做复杂并发调度、抢占或自动负载均衡
- 不在本次变更中重写所有提示词资产

## Decisions

### 1. 使用“主 agent + 专职 agent”分层，而不是完全对等 mesh

原因：

- 现有产品形态仍以单用户单任务入口为主
- 主 agent 负责理解用户意图和拆分任务，专职 agent 负责局部执行，更符合当前使用方式
- 比完全对等 agent mesh 更容易控制上下文和工具权限

### 2. agent 配置外置化，但执行内核复用现有 AI core

原因：

- 需要独立定义 agent 名称、职责、工具白名单、默认上下文策略
- 现有 `stream_agent_with_tools`、React 协议等执行核心已有可复用价值
- 这样可以先解决“角色定义与路由”问题，而不是重写底层推理流

### 3. 委派结果采用显式 task/result 协议，不共享隐式长上下文

原因：

- 当前系统最缺的是可解释的任务边界
- 若子 agent 直接共享全部长上下文，会让调试和权限控制重新失效
- 显式协议更适合未来与工具规范化、上下文可视化联动

### 4. 一期先接入写作助手、关系抽取、审校三类 agent

原因：

- 这三类职责已经在系统中有明显影子
- 能够覆盖创作、分析、审阅三种典型场景
- 范围足够清晰，不会把全部抽取器一次性迁入

## Risks / Trade-offs

- [风险] 主 agent 再次变成“超级调度器”，提示词依然过大  
  → Mitigation：限制主 agent 只负责拆分与汇总，不直接承担所有专业判断

- [风险] 现有单 agent 与多 agent 并存一段时间，会增加维护成本  
  → Mitigation：用 capability 和路由配置明确哪些入口先接入，分阶段迁移

- [风险] 子 agent 结果结构不统一会导致整合困难  
  → Mitigation：定义标准 task/result 契约，避免自由文本回传

- [风险] 用户可能难以理解“为什么切到了另一个 agent”  
  → Mitigation：后续与上下文可视化/执行日志联动，展示委派记录

## Migration Plan

1. 定义 agent 配置模型与注册机制
2. 建立 task/result 协议与委派执行流
3. 先让写作助手在部分任务下路由到专职 agent
4. 再逐步接入关系抽取和审校链路
5. 最后评估是否替换更多单入口 AI 调用

回滚策略：

- 可通过关闭多 agent 路由配置退回单 agent 模式
- 不改数据库核心结构的前提下，回滚主要在服务层与配置层完成

## Open Questions

- agent 配置应先落在数据库、文件系统，还是混合模式？
- 是否需要允许用户手动选择 agent，还是一期只做自动路由？
- 子 agent 失败时，主 agent 是否允许接管重试，还是直接显式报错？
